\title{SurtGIS: A High-Performance Raster Geospatial Analysis Library in Rust with WebAssembly and Python Support}

\author[usach]{Francisco Parra\corref{cor1}}
\ead{francisco.parra.o@usach.cl}
\cortext[cor1]{Corresponding author}

\affiliation[usach]{organization={Departamento de Ingeniería Informática, Universidad de Santiago de Chile},%
            city={Santiago},%
            country={Chile}}

\begin{abstract}
We present SurtGIS, an open-source raster geospatial analysis library in Rust providing 127 terrain, hydrological, and remote sensing algorithms. The library compiles to native code with automatic parallelization via Rayon, to WebAssembly for browser-based applications, and exposes Python bindings through PyO3. Key contributions include Florinsky's complete 14-curvature morphometric system, three depression handling algorithms, and four flow direction methods. Benchmarks on synthetic DEMs up to $20{,}000^2$ cells demonstrate speedups on full GeoTIFF pipelines: 1.7--1.8$\times$ over GDAL and 4.5--4.9$\times$ over GRASS~GIS for slope; 2.0$\times$ over GDAL for aspect; and 7.5--23.1$\times$ for D8 flow accumulation. WebAssembly adds 10--32$\times$ overhead versus native Rust but enables browser-based terrain analysis unavailable in existing tools. Numerical accuracy is within $5.7 \times 10^{-5}$~degrees RMSE against analytical solutions.
\end{abstract}

\section{Introduction}
\label{sec:introduction}

Digital terrain analysis is fundamental to environmental modeling, encompassing slope stability assessment, hydrological network extraction, landform classification, and habitat suitability mapping \citep{florinsky2025}. The increasing availability of high-resolution elevation data---from lidar surveys at sub-meter resolution to global datasets such as SRTM \citep{farr2007srtm} and Copernicus DEM at 30\,m---has created demand for efficient, accessible, and reproducible geospatial processing tools.

Current geospatial software presents researchers with a trade-off between performance and accessibility. Python-based tools such as rasterio and richdem offer ease of use but are constrained by interpreter overhead for pixel-level computation. Compiled libraries such as GDAL \citep{gdal2024} provide performance but lack comprehensive terrain analysis algorithms and cannot execute in browser environments. Specialized tools such as WhiteboxTools \citep{lindsay2016wbt} and GRASS GIS \citep{neteler2012} offer extensive algorithm collections but are either partially proprietary or require heavyweight installations unsuitable for web deployment. RichDEM \citep{barnes2016richdem} provides efficient C++ implementations of hydrological algorithms with Python bindings, SAGA GIS \citep{conrad2015} offers a comprehensive suite of terrain analysis modules, and TauDEM \citep{tarboton2015taudem} provides the reference D-infinity flow implementation \citep{tarboton1997} via MPI-based parallelism. In the Python ecosystem, pysheds \citep{bartos2020pysheds} and pyflwdir \citep{eilander2021pyflwdir} provide lightweight hydrological routing but lack terrain morphometry and browser deployment. \revtext{TauDEM is included in the hydrological benchmarks as the representative MPI-based parallel geocomputation framework, executed on the same machine across 1--16 MPI processes (Section~\ref{subsec:taudem_mpi}).}

Cloud-native geospatial formats---Cloud-Optimized GeoTIFF (COG), SpatioTemporal Asset Catalogs \citep[STAC;][]{stac2021}, and Zarr---and the platforms that publish them \citep{gorelick2017gee} have shifted the dominant access pattern from full-file download to on-demand spatial-range reads over HTTP. This benefits any client capable of issuing HTTP byte-range requests: batch processing services, scientific workflows, and increasingly client-side applications running in browsers. The browser case in particular requires libraries that can execute computations on the client without round-tripping data to a server, which motivates WebAssembly (Wasm) as a compilation target \citep{haas2017}; existing comprehensive raster analysis libraries do not provide a Wasm build.

We present SurtGIS, an open-source raster geospatial analysis library implemented in Rust that addresses these limitations. Rust is the implementation language because it satisfies three constraints simultaneously: (i) memory safety without garbage collection \citep{perkel2020rust}, allowing predictable latency and cache-friendly memory layout that interpreted languages cannot guarantee; (ii) zero-cost abstractions yielding C-level performance for tight numerical loops; and (iii) first-class WebAssembly compilation, which enables identical algorithms to run client-side without the runtime overhead and fork-based parallelism that CPython requires for shared-memory work. Python users access the same engine without rewriting workflows: the \texttt{surtgis-python} package is a PyO3 binding over the Rust core, installable via \texttt{pip install surtgis} so that the speedups reported here apply to Python notebook usage at the cost of a single zero-copy buffer view at the language boundary. The library provides:

\begin{enumerate}
    \item 127 distinct computational operations for terrain analysis, hydrology, remote sensing, landscape ecology, and texture analysis (complete inventory and counting methodology in Supplementary Table~S1);
    \item An open-source implementation of Florinsky's complete 14-curvature morphometric system \citep{florinsky2025};
    \item A unified codebase that compiles to native (with Rayon parallelization), WebAssembly (sequential fallback), and Python (via PyO3);
    \item Cloud-native data access through integrated STAC client and COG reader.
\end{enumerate}

This paper is structured as follows: Section~\ref{sec:software} describes the software architecture and algorithms; Section~\ref{sec:example} provides an illustrative example; Section~\ref{sec:casestudy} presents an application example; Section~\ref{sec:performance} presents the performance evaluation; Section~\ref{sec:comparison} compares with existing tools; and Section~\ref{sec:conclusions} discusses impact and conclusions.

\subsection{Architecture}
\label{subsec:architecture}

SurtGIS is organized as a Rust workspace of nine crates organised in three layers (Figure~\ref{fig:architecture}): an I/O layer (\texttt{surtgis-core}: raster data structures, native GeoTIFF I/O, NoData handling; \texttt{surtgis-cloud}: STAC client, COG reader, Zarr/NetCDF/GRIB readers behind feature flags), a compute layer (\texttt{surtgis-algorithms}: all geospatial algorithms; \texttt{surtgis-parallel}: Rayon-based parallel adapters; \texttt{surtgis-colormap}: color schemes and raster rendering), and a bindings layer (\texttt{surtgis-cli}: command-line interface; \texttt{surtgis-wasm}: WebAssembly bindings; \texttt{surtgis-python}: PyO3 bindings; \texttt{surtgis-gui}: optional egui desktop application, not benchmarked here). This modular structure allows users---and downstream binaries---to depend only on the layers they need: a Python notebook user pulls in \texttt{surtgis-python}, which in turn pulls \texttt{surtgis-algorithms} and \texttt{surtgis-core}, with no runtime dependency on the cloud or GUI crates.

The raster data model stores elevation and derived grids as contiguous \texttt{ndarray::Array2<f64>} arrays with associated metadata (cell size, NoData value, spatial reference, affine transform). This in-memory representation enables efficient cache-friendly access patterns for moving-window operations that dominate terrain analysis workloads, though it requires holding the full dataset in RAM (see Section~\ref{subsec:scalability} for memory implications).

\paragraph{Data robustness} All algorithms propagate NoData values: boundary cells and neighbors of NoData pixels produce NoData in the output rather than artifacts. Non-square pixels (different $\Delta x$ and $\Delta y$) are supported via the affine transform metadata; slope and curvature computations use per-axis cell sizes. Input validation uses Rust's \texttt{Result} type throughout the API, with explicit error types for invalid rasters (empty, non-finite values), unsupported CRS, and I/O failures. \textbf{Important:} geographic CRS inputs (e.g., WGS84 in degrees) are currently processed in degree units, which produces physically meaningless slope and curvature values; users must reproject to a metric CRS (e.g., UTM) before analysis. NaN and infinity values in input rasters are treated as NoData.

\subsection{Parallel strategies}
\label{subsec:parallel_strategies}

SurtGIS targets single-node throughput with cloud-native streaming I/O, a complementary point in the design space to distributed-memory raster libraries such as pRPL \citep{guan2010prpl}, pRPL+GTIOL \citep{miao2017prplgtiol}, and the parallel raster-based geocomputation operator frameworks of \citet{qin2014strategy}, which target cluster execution under the Message Passing Interface (MPI). \revtext{For precision of terminology: MPI is a message-passing standard and library ecosystem for distributed-memory programming, whereas massively parallel processing (MPP) denotes a class of system architectures on which such libraries commonly run; the two belong to different technical tiers and are not used interchangeably here.} Single-node decomposition is appropriate for the deployment surfaces SurtGIS targets---desktop and laptop machines, single virtual machines, web browsers via WebAssembly, and Python notebooks---because the data volumes those surfaces can accommodate (DEMs up to $20\text{K}^2$ in our benchmarks, individual Sentinel-2 scenes, single watersheds) are tractable on one machine when paired with streaming GeoTIFF I/O and HTTP range reads against cloud object storage. Distributed-memory parallelism via MPI is not displaced by SurtGIS but is a distinct deployment regime; the strategies described below characterise within-node decomposition, scheduling, and aggregation\revtext{, and Section~\ref{subsec:taudem_mpi} provides a direct single-node benchmark against TauDEM as the MPI representative}.

\subsubsection{Decomposition patterns}

\paragraph{Domain decomposition} The default partition is by row strip: a Rayon \texttt{par\_iter} over chunks of the row index splits an $H \times W$ grid into roughly $\lceil H/p \rceil$-row strips per worker, where $p$ is the number of threads. Row strips are preferred over square tiles for three reasons: (i) GeoTIFF files store data row-wise on disk, so row-aligned access patterns minimise seeks and maximise sequential I/O bandwidth, mirroring the design rationale of pRPL+GTIOL \citep{miao2017prplgtiol}; (ii) for focal kernels with halo $k$ (typically $k=1$ for Horn-method slope and aspect, or $k=10$ for radius-10 TPI), a row strip requires only $2k$ extra rows per worker for boundary reads, whereas square tiles require $O(k\sqrt{N/p})$ halo cells; (iii) row strips reuse memory in contiguous f64 array slices, which the Rust compiler and the CPU prefetcher can vectorise without explicit SIMD intrinsics.

\paragraph{Result aggregation} For cell-wise and focal patterns, each worker writes into a disjoint slice of the output buffer; no aggregation is needed and no synchronisation primitive is acquired during the parallel section. For global-reduction algorithms, Rayon's \texttt{reduce} combinator merges per-strip partials in $O(\log p)$ tree time after the parallel map phase. Dependency-aware algorithms (Priority-Flood, D8 flow accumulation) cannot be fully domain-decomposed without changing the per-cell algorithm; the v0.7 release uses serial or partially-parallel implementations whose Amdahl-bound behaviour is empirically characterised in Section~\ref{subsec:parallel_scaling}. Tile-and-reconcile parallel variants of \citet{barnes2017} are on the SurtGIS roadmap but are not part of this evaluation.

\paragraph{Inter-thread communication} The decomposition is structured so that workers do not communicate during the parallel section. Halo rows are read-only: the input array is shared via Rust's borrow checker as an immutable reference, and each worker reads from the input strip plus its halo without ever writing to a region another worker reads. The output buffer is accessed via mutually disjoint mutable slices (\texttt{split\_at\_mut}), an aliasing-free pattern verified at compile time. There are no atomic operations, mutexes, or message passes inside the inner loops of any cell-wise or focal kernel. This contrasts with shared-mutable patterns common in OpenMP raster code, where atomic increments on a shared accumulator can dominate at high $p$.

\paragraph{Task scheduling and load balancing} Rayon scheduling uses work-stealing \citep{blumofe1999}: each worker has a deque of tasks and idle workers steal from busy peers. SurtGIS does not pin threads to cores, leaving placement to the OS scheduler; on the hybrid Intel i7-1270P used in our benchmarks (4 P-cores + 8 E-cores), this means performance and efficiency cores receive work according to availability rather than a fixed split. For uniform-work patterns (cell-wise and focal), the row-strip decomposition assigns each worker an equal cell count, so static load balance is intrinsic to the partition and the work-stealing layer absorbs only the residual variance from OS interrupts and cache effects. Section~\ref{subsec:parallel_scaling} confirms this empirically: focal-kernel efficiency curves do not exhibit imbalance signatures (sharp drops at specific $p$), only the smooth bandwidth-bound saturation expected from a well-balanced memory-bound workload.

\paragraph{Geocomputational heterogeneity} Non-uniform per-cell work can arise when (a) cells contain NoData and short-circuit the inner kernel, (b) flow-routing algorithms have data-dependent recursion depth, or (c) viewshed and visibility computations have variable ray cardinalities depending on local relief. For (a), the kernel cost of skipping a NoData cell is approximately $10\%$ of a full kernel, so the worst-case imbalance between a strip with $100\%$ NoData and a strip with $100\%$ data is $\sim 10\times$; Rayon's dynamic chunk sizing absorbs this by re-distributing fragments of strips to idle workers. For (b) and (c), SurtGIS uses task-level rather than data-level parallelism (e.g., viewshed-from-many-points dispatches one Rayon task per observer; multi-scale curvature dispatches one task per scale), recovering near-linear scaling whenever the algorithm admits multiple independent task instances. The cell-level partition is only chosen when the per-cell work is approximately uniform.

\paragraph{\revtext{Static decomposition versus dynamic work-stealing}} \revtext{The parallel geocomputation literature has developed sophisticated \emph{static} domain decomposition strategies---spatially coherent aggregation of adjacent subtasks and complexity-driven partition sizing informed by per-region workload estimates \citep{qin2014strategy, wang2021loadbalancing}. These strategies are essential in the distributed-memory (MPI) setting for which they were designed: the initial decomposition there is fixed for the lifetime of the computation, migrating work between processes is expensive, and a poor initial partition therefore determines the achievable load balance. A shared-memory work-stealing runtime occupies a different point in this design space. Rayon divides the domain into sub-ranges substantially finer than the worker count, and idle threads steal queued sub-ranges from busy peers at a cost of nanoseconds rather than message-passing latencies; the load-balancing problem that complexity-driven static partitioning solves \emph{a priori} from properties of the input raster is thus solved \emph{at runtime} by the scheduler---including for imbalance sources that no static analysis of the input can predict, such as data-dependent branch costs and interference from other processes on the machine. The two approaches are therefore complements attached to different memory models rather than competing alternatives at the same tier: for the reprojection-corner NoData pattern---the worst case for a static row-strip partition, in which entire strips near a rotated grid's edges are dominated by short-circuiting NoData cells---the work-stealing scheduler redistributes the surviving sub-ranges to idle threads at runtime, so no static, complexity-aware partition of the input is required to keep cores busy. For globally coupled algorithms (Priority-Flood, watershed labelling, flow accumulation), where neither fine-grained stealing nor static partitioning applies directly, decomposition is genuinely harder; the static tile-and-reconcile strategies of the cited literature remain the appropriate foundation for a future distributed-memory tier of SurtGIS (Section~\ref{sec:conclusions}).}

\paragraph{Streaming} For DEMs that exceed available RAM, twelve terrain algorithms implement an out-of-core streaming variant that reads, computes, and writes a single strip at a time, retaining only $2k$ halo rows between iterations (where $k$ is the kernel radius). The streaming variant uses the same row-strip parallel decomposition within each loaded strip, so the speedup characterisation in Section~\ref{subsec:parallel_scaling} carries through. The disk I/O itself is serial in v0.7 (read-then-compute-then-write per strip); concurrent read-while-compute pipelining is identified as future work.

\subsubsection{Slope and aspect}

SurtGIS computes slope and aspect using Horn's method \citep{horn1981}, which estimates first-order partial derivatives from a $3 \times 3$ moving window using a weighted finite-difference scheme. The slope gradient is:

\begin{equation}
\text{slope} = \arctan\!\sqrt{\left(\frac{\partial z}{\partial x}\right)^2 + \left(\frac{\partial z}{\partial y}\right)^2}
\label{eq:slope}
\end{equation}

\noindent where the partial derivatives are estimated by the Horn (1981) $3 \times 3$ kernel:
\begin{equation}
\frac{\partial z}{\partial x} = \frac{(z_{+1,-1} + 2z_{+1,0} + z_{+1,+1}) - (z_{-1,-1} + 2z_{-1,0} + z_{-1,+1})}{8 \Delta x}
\label{eq:horn_kernel}
\end{equation}
\noindent (and analogously for $\partial z / \partial y$), using weights $[1, 2, 1]$ for the central row/column. This is the same method used by GDAL and GRASS GIS. WhiteboxTools uses a 3rd-order bivariate Taylor polynomial on a $5 \times 5$ neighborhood \citep{florinsky2016}, which is more robust against noise but produces systematically different slope estimates than Horn's $3 \times 3$ kernel; this methodological difference accounts for the cross-tool deviations reported in Section~\ref{subsec:accuracy}.

\subsubsection{Florinsky's 14-curvature system}

A key contribution of SurtGIS is the open-source implementation of the complete morphometric variable system described by \citet{florinsky2025}. Surface curvatures are computed from partial derivatives estimated via the Evans--Young method \citep{evans1972} on a $3\times3$ moving window. A local bivariate polynomial $z = rx^2/2 + sxy + ty^2/2 + px + qy + z_0$ is fitted by least squares to the nine elevation values in the window, yielding the first-order ($p = \partial z/\partial x$, $q = \partial z/\partial y$) and second-order ($r = \partial^2 z/\partial x^2$, $s = \partial^2 z/\partial x \partial y$, $t = \partial^2 z/\partial y^2$) partial derivatives. From these, the 14 curvature types are derived:

\begin{equation}
H = -\frac{(1+q^2)r - 2pqs + (1+p^2)t}{2(1+p^2+q^2)^{3/2}}
\label{eq:mean_curv}
\end{equation}

\begin{equation}
K = \frac{rt - s^2}{(1+p^2+q^2)^2}
\label{eq:gauss_curv}
\end{equation}

\noindent where $H$ is mean curvature and $K$ is Gaussian curvature. Note that the sign convention for $H$ (Eq.~\ref{eq:mean_curv}) follows \citet{florinsky2025}: the negative sign in the numerator yields $H > 0$ for convex surfaces (ridges) and $H < 0$ for concave surfaces (valleys). Some references omit this negation, producing the opposite convention; users should verify sign semantics when comparing across tools. The remaining 12 curvatures (unsphericity $M$, difference $E$, principal curvatures $k_{\min}$ and $k_{\max}$, horizontal $k_h$, vertical $k_v$, excess curvatures $k_{he}$ and $k_{ve}$, accumulation $K_a$, ring $K_r$, rotor, and Laplacian) are derived from combinations of $H$, $K$, and the partial derivatives. SAGA GIS implements a subset (profile, plan, tangential, longitudinal, minimal, maximal curvature) based on earlier Florinsky work, but does not include the excess curvatures ($k_{he}$, $k_{ve}$), accumulation ($K_a$), ring ($K_r$), unsphericity ($M$), or difference ($E$) curvatures from the 2025 formulation. A comprehensive taxonomy of surface curvatures and their physical interpretation is provided by \citet{minar2020}. To our knowledge---based on a survey of GDAL, GRASS, SAGA, WhiteboxTools, RichDEM, and TauDEM documentation as of February 2026---SurtGIS is the first open-source library to implement the complete 14-type system as defined by \citet{florinsky2025}. We acknowledge that the book's recent publication date (2025) means other implementations may emerge.

\subsubsection{Geomorphons}

The geomorphon classification algorithm \citep{jasiewicz2013} identifies landform elements by analyzing the ternary pattern of elevation differences along eight cardinal and diagonal directions within a search radius. SurtGIS implements this with configurable flatness threshold and lookup radius, producing a 10-class landform map (flat, peak, ridge, shoulder, spur, slope, hollow, footslope, valley, pit).

Depression handling is a critical preprocessing step. SurtGIS implements three approaches: (1) the Planchon--Darboux filling algorithm \citep{planchon2001}, which produces flat areas that can affect flow routing; (2) the Priority-Flood algorithm \citep{barnes2014}, an optimal $O(N \log N)$ method using a priority queue; and (3) the breach-based approach of \citet{lindsay2016}, which carves channels through barriers to maintain more natural flow paths.

For flow direction, the D8 algorithm \citep{ocallaghan1984} assigns flow to the single steepest downslope neighbor. The Multiple Flow Direction (MFD) algorithm of \citet{quinn1991} distributes flow proportionally to all downslope neighbors based on slope, better representing sheet flow on gentle terrain. The D-infinity algorithm of \citet{tarboton1997}, implemented in TauDEM \citep{tarboton2015taudem}, uses triangular facets to determine a continuous flow angle, partitioning flow between the two cells that bracket this angle.

\subsection{Additional modules}
\label{subsec:additional}

Beyond terrain and hydrology, SurtGIS includes modules for remote sensing (22 algorithms: 16 spectral indices including NDVI, NDWI, EVI, BSI, NBR, plus band-math, reclassify, raster difference, change-vector analysis), landscape ecology (Shannon and Simpson diversity, patch density), texture analysis (GLCM Haralick features and Sobel edge detection), classification (PCA, $k$-means, ISODATA, minimum distance, maximum likelihood), morphology (erosion, dilation, opening, closing, gradient, top-hat, black-hat), interpolation (IDW, nearest neighbour, TIN, natural neighbour, thin-plate spline, ordinary / universal / regression kriging), and statistics (focal stats, zonal stats, Moran's $I$, Getis--Ord $G_i^*$). These modules share the same cross-platform compilation model but are not the focus of the benchmarks presented here; a complete inventory of all 127 algorithms is provided in Supplementary Table~S1.

\subsection{Cross-platform deployment}
\label{subsec:deployment}

SurtGIS compiles to three deployment targets from the same codebase:

\paragraph{Native (desktop/server)} Compiled with Rayon for automatic data parallelism. Published as Rust crates on \url{https://crates.io}.

\paragraph{Python} Bindings via PyO3 \citep{pyo3_2024} expose 97 functions covering terrain, hydrology, imagery, morphology, statistics, interpolation, classification, landscape, and texture with NumPy array input/output. Functions are organized under a flat \texttt{surtgis} namespace (e.g., \texttt{surtgis.slope()}, \texttt{surtgis.advanced\_curvature()}); a submodule structure (\texttt{surtgis.terrain}, \texttt{surtgis.hydrology}) is planned for a future release. Data transfer uses NumPy's buffer protocol for zero-copy access where possible; when the Rust function requires owned data, a single \texttt{memcpy} is performed. Published on PyPI as the \texttt{surtgis} package, installable via \texttt{pip install surtgis}. Prebuilt wheels are produced by the \texttt{python-release.yml} GitHub Actions workflow for Linux (x86\_64, aarch64; manylinux and musllinux), macOS (x86\_64, ARM64), and Windows (x86\_64) across CPython 3.9--3.13. The remaining 30 algorithms are progressively exposed in each release, with full Python coverage targeted for version~1.0; all 127 are accessible via the native Rust API.

\paragraph{WebAssembly} Compiled via \texttt{wasm-pack} with JavaScript/TypeScript bindings. Published on npm as the \texttt{surtgis} package. Enables terrain analysis directly in web browsers without server-side processing.

\section{Illustrative example}
\label{sec:example}

To demonstrate SurtGIS's capabilities, we present a terrain characterization workflow in Python. The example synthesizes a small DEM in NumPy, computes terrain derivatives, and extracts hydrological features. The synthetic DEM is fully reproducible (fixed random seed) so the listing runs end-to-end without external data dependencies:

\begin{verbatim}
import numpy as np
import surtgis

# Synthesize a 200x200 DEM at 30 m resolution: a Gaussian
# hill superimposed on multi-octave deterministic noise.
n, cell_size = 200, 30.0
y, x = np.mgrid[0:n, 0:n] * cell_size
hill = 1500 * np.exp(-((x - n*cell_size/2)**2 +
                       (y - n*cell_size/2)**2) / (1500**2))
rng = np.random.default_rng(seed=42)
noise = sum((1.0 / 2**k) * rng.standard_normal((n, n))
            for k in range(1, 5))
dem = (hill + 200 * noise).astype(np.float64)

# Terrain derivatives
slope = surtgis.slope(dem, cell_size)
aspect = surtgis.aspect_degrees(dem, cell_size)
hillshade = surtgis.hillshade_compute(
    dem, cell_size, azimuth=315.0, altitude=45.0)

# Florinsky curvatures
mean_curv = surtgis.advanced_curvature(
    dem, cell_size, ctype="mean_h")
gaussian_curv = surtgis.advanced_curvature(
    dem, cell_size, ctype="gaussian_k")

# Hydrology pipeline
filled = surtgis.priority_flood_fill(dem, cell_size)
twi = surtgis.twi_compute(dem, cell_size)
hand = surtgis.hand_compute(dem, cell_size)
streams = surtgis.stream_network_compute(
    dem, cell_size, threshold=100.0)
\end{verbatim}

The same algorithms are available in the browser through the WebAssembly bindings, enabling interactive web applications where users can upload a DEM and compute terrain derivatives client-side without transmitting data to a server.

\paragraph{Cloud-native end-to-end workflow} A second representative usage scenario is the cloud-native pipeline that exercises the \texttt{surtgis-cloud} crate together with the algorithms layer. From the command line, three invocations take the user from a STAC catalogue query to a derived spectral index, with no intermediate Python or shell glue (Figure~\ref{fig:workflow}): (i)~\texttt{surtgis stac composite} searches a STAC endpoint (Microsoft Planetary Computer or Earth Search), downloads the matching scenes via HTTP range reads against the source COG assets, applies the catalogue's cloud-mask convention (SCL for Sentinel-2 L2A or QA\_PIXEL for Landsat) to each scene, and reduces the temporal stack to a per-band median composite written as GeoTIFF; (ii)~\texttt{surtgis imagery ndvi} computes the normalized difference vegetation index from the red and near-infrared composite bands; (iii)~\texttt{surtgis info} prints summary statistics for verification. On the test hardware, this end-to-end pipeline runs in approximately 65\,s for a 5$\times$5\,km bounding box (one month of Sentinel-2 imagery, three cloud-free scenes, two output bands) over a typical residential connection, and the full pipeline is reproducible from a single command-line block included in the project README.

\section{Application example: Andean terrain characterization}
\label{sec:casestudy}

To illustrate SurtGIS's workflow capabilities, we present a terrain characterization of a mountainous region in the central Chilean Andes (approximately $33.5^{\circ}$S, $70.3^{\circ}$W), using a Copernicus DEM 30\,m tile reprojected to UTM zone~19S ($662 \times 785$ cells, cell size $= 28.4$\,m). The study area spans elevations from 2{,}859\,m to 5{,}979\,m and covers approximately $19 \times 22$\,km of rugged terrain characterized by deeply incised valleys and steep ridge systems. We chose this high-relief Andean setting because it exercises SurtGIS's full range of terrain and hydrological algorithms---steep slopes, high curvature contrasts, deep depressions, and a well-defined drainage network---rather than for operational flood mapping, which would require flatter, lower-elevation terrain and observed inundation data for calibration. This type of multi-layer terrain characterization is commonly required for landslide susceptibility mapping, hydrological modeling, and habitat characterization \citep{amatulli2018}.

SurtGIS computes eight analysis layers from the DEM in a single Python session (Figure~\ref{fig:casestudy}): slope, aspect, hillshade, mean curvature ($H$), geomorphon classification, depression-filled surface, Topographic Wetness Index (TWI), and Height Above Nearest Drainage (HAND) with stream network extraction. The entire workflow---including GeoTIFF I/O---executes in 0.3\,s on the test hardware (0.5\,M cells; for performance on larger DEMs, see Section~\ref{sec:performance}).

\paragraph{Terrain morphology} The mean slope of 29\textdegree{} confirms the rugged character of the study area (Figure~\ref{fig:casestudy}b). Florinsky's mean curvature $H$ (Figure~\ref{fig:casestudy}c) reveals the ridge-valley structure: positive values (red) mark convex ridges and spurs where erosion dominates, while negative values (blue) identify concave hollows and channels where deposition occurs. The curvature field provides information beyond slope alone, as it distinguishes planar slopes from convergent and divergent landforms. Access to the full 14-curvature system enables further characterization: horizontal curvature $k_h$ identifies zones of flow convergence (concavity in the plan view) relevant to landslide initiation, while the difference curvature $E$ quantifies local flow asymmetry between principal directions. These additional curvatures are computed in the same Evans--Young derivative pass as $H$ and $K$, adding negligible overhead to the workflow.

\paragraph{Landform classification} The geomorphon algorithm \citep{jasiewicz2013}, applied with a lookup radius of 10 cells ($\sim$284\,m) and a flatness threshold of 1\textdegree, classifies the study area into three dominant landform types (Figure~\ref{fig:casestudy}d): slope (44.9\%), hollow (27.1\%), and spur (26.5\%), with minor contributions from peaks (0.9\%) and valleys (0.2\%). The near-absence of flat terrain is consistent with the high-gradient Andean environment. These proportions are sensitive to the lookup radius parameter: smaller radii detect finer-scale features (increasing the proportion of slope-class cells), while larger radii generalize the classification toward broader landform categories \citep{jasiewicz2013}. This automated classification provides a parameter-parsimonious alternative to manual terrain unit delineation.

\paragraph{Hydrological analysis} The TWI \citep{kopecky2021} identifies zones of potential soil saturation in valley bottoms (Figure~\ref{fig:casestudy}e), while HAND values \citep{nobre2011} delineate the vertical distance to the nearest drainage channel (Figure~\ref{fig:casestudy}f). The extracted stream network (blue overlay in Figure~\ref{fig:casestudy}f, $\sim$32{,}000 stream cells) follows the dendritic drainage pattern typical of this geological setting. HAND values reach 1{,}698\,m on the highest ridges, providing a continuous indicator of flood susceptibility that is increasingly used in operational flood mapping.

\paragraph{Flood susceptibility mapping} To demonstrate an applied environmental analysis, we use HAND thresholds from \citet{nobre2011} (calibrated for Amazonian floodplains; illustrative for this Andean setting) to delineate flood susceptibility zones (Figure~\ref{fig:floodzones}): high risk (HAND $< 5$\,m, 3.4\%), moderate (5--10\,m, 1.7\%), low (10--15\,m, 1.5\%), minimal (15--25\,m, 2.7\%), and negligible ($> 25$\,m, 85.9\%); the remaining $\sim$4.8\% corresponds to NoData boundary cells. The combined high-to-moderate risk zone covers 5.1\% of the study area, concentrated along valley bottoms. We note that HAND values are sensitive to the stream extraction threshold (100 cells, $\sim$0.08\,km$^2$): varying between 50--200 cells shifts the HAND distribution, though the large-scale spatial pattern is preserved. These results are presented as a workflow demonstration, not an operational flood hazard map.

\paragraph{Integration} The derived layers form a feature stack suitable for statistical modeling: slope, curvatures, and TWI are standard predictors in logistic regression or random forest models for landslide susceptibility \citep{amatulli2018}. The key practical advantage is that SurtGIS computes all required layers from a single DEM in a unified workflow, eliminating the need to chain GDAL (for basic derivatives), GRASS (for hydrology), and manual scripting to integrate outputs with potentially incompatible data formats. For instance, a multi-hazard assessment on a $10\text{K}^2$ DEM (100\,M cells) requiring slope, curvatures, TWI, and HAND would complete in under 60\,s with SurtGIS, compared to several minutes when orchestrating separate GDAL and GRASS invocations with intermediate file transfers. The unified API also ensures spatial metadata consistency across all derived layers, avoiding the CRS and extent mismatches that frequently arise when chaining heterogeneous tools. This section demonstrates SurtGIS as a workflow tool; validation of specific environmental applications (e.g., calibrated flood mapping, landslide susceptibility modeling) against field observations is deferred to domain-specific studies.

\section{Performance evaluation}
\label{sec:performance}

We evaluate SurtGIS performance through four experiments: (1) a scalability benchmark comparing execution time against established tools across DEM sizes; (2) a numerical accuracy validation against analytical and cross-tool references; (3) a cross-platform comparison between native Rust, Python bindings, and single-threaded execution; and (4) a parallel-scaling analysis quantifying speedup, parallel efficiency, and the regimes that limit each algorithm class.

All benchmarks were executed on a single machine: Intel Core i7-1270P (4 performance cores up to 4.8\,GHz + 8 efficiency cores up to 3.5\,GHz, 16 threads), \SI{38.8}{\giga\byte} DDR4 RAM, Ubuntu 24.04 LTS (kernel 6.14.0). The CPU governor was set to \texttt{powersave} (the system default, which allows the kernel to dynamically scale frequency); Intel Turbo Boost was enabled. We retained the default governor rather than \texttt{performance} because: (a)~it reflects the out-of-the-box configuration users are most likely to encounter, and (b)~all tools are benchmarked under the same conditions, so the relative speedups remain valid even if absolute times vary with governor policy. Tool versions: GDAL~3.11.4, GRASS~GIS~8.3.2, WhiteboxTools~2.4.0, SurtGIS~0.1.1 (commit \texttt{7493bc1}). The benchmark automation script (\texttt{benchmarks/run\_benchmarks.py}), the native Rust benchmark (\texttt{bench\_comparison.rs}), the WASM benchmark (\texttt{bench\_wasm.mjs}), and all raw CSV results are included as supplementary material. DEM generation uses NumPy~2.4 with a fixed random seed (\texttt{seed=42}) for full reproducibility. \revtext{The load-balancing experiment of Section~\ref{subsec:nodata_heterogeneity} and the cross-platform MPI check of Section~\ref{subsec:taudem_mpi} additionally use a secondary platform with homogeneous cores and stable clocks---a Hetzner CCX33 instance (8 dedicated AMD EPYC-Milan cores at 2.0\,GHz, 32\,GB RAM, Ubuntu 24.04)---chosen because its uniform cores isolate load-balancing behaviour from the hybrid-core and thermal-throttling effects of the primary machine.}

\paragraph{Versioning} Experiments~1--3 were performed on SurtGIS~0.1.1 (commit \texttt{7493bc1}) and reported here unchanged. The parallel-scaling analysis (Experiment~4, Section~\ref{subsec:parallel_scaling}) was performed on the current development tree, released as v0.7.0. The 127 algorithms benchmarked are unchanged across these versions: the v0.6.x and v0.7.0 releases added cloud-native I/O (STAC and COG readers), out-of-core streaming variants for twelve terrain algorithms (Section~\ref{subsec:parallel_strategies}), Python and CLI tooling for CNN patch extraction, and a documentation site, none of which modify the per-cell algorithm or alter the parallel-decomposition strategies analysed in this paper. To verify continuity we re-ran the Experiment~1 SurtGIS measurements on v0.7.0 ($n=10$ per cell, same DEMs, same harness): v0.7.0 produces equal or faster full-pipeline times across all six algorithms at both $5\text{K}^2$ and $10\text{K}^2$, with reductions of 4--61\,\%. The speedup ratios reported in Tables~\ref{tab:slope_results}--\ref{tab:all_algos} are therefore conservative on the current release. The pre-1.0 development policy is to pre-announce breaking API changes one release ahead under a \emph{Breaking} heading in the changelog; no breaking change has occurred between v0.1.1 and v0.7.0 affecting the algorithms benchmarked here.

The exact command lines used for each tool are:
\begin{itemize}
    \item \textbf{SurtGIS}: \texttt{cargo run --example bench\_comparison --release -{-} -{-}size N -{-}algo slope}
    \item \textbf{GDAL}: \texttt{gdaldem slope input.tif output.tif -of GTiff -b 1} (default Horn algorithm, \texttt{-compute\_edges} not used to match SurtGIS boundary handling)
    \item \textbf{GRASS}: \texttt{grass --tmp-location EPSG:32719 --exec bash -c "r.in.gdal input=... output=dem; r.slope.aspect elevation=dem slope=out; r.out.gdal input=out output=..."}
    \item \textbf{WBT}: \texttt{whitebox\_tools -r=Slope -v --dem=input.tif -o=output.tif}
\end{itemize}

\subsection{Experiment 1: Scalability}
\label{subsec:scalability}

We compared wall-clock execution time for six algorithms against GDAL~3.11, GRASS~GIS~8.3, and WhiteboxTools~2.4 on synthetic DEMs of increasing size: $1\text{K}^2$ (1\,M cells), $5\text{K}^2$ (25\,M), $10\text{K}^2$ (100\,M), and $20\text{K}^2$ (400\,M cells). Synthetic DEMs were generated using a fractal Brownian surface (Hurst exponent $= 0.7$, fixed seed for reproducibility) to ensure identical input data across all tools.

The algorithms tested are: (a) slope, (b) aspect, (c) hillshade, (d) TPI, (e) Priority-Flood depression filling, and (f) D8 flow accumulation. These cover both simple moving-window operations (a--d) and graph-based algorithms with different computational complexity (e--f).

Each measurement was repeated 10 times after 3 warm-up runs. We report median execution time; interquartile range (IQR) is shown in the scalability figures and reported where relevant. The coefficient of variation (CV) was below 10\% for all reported measurements. Across DEM sizes, per-algorithm median CV ranged from 0.8\% (fill at 1\,K$^2$) to 4.1\% (slope at 20\,K$^2$); the single highest individual CV was 6.4\% (slope at $20\text{K}^2$, one run). Given the sample sizes ($n=10$ for scalability experiments, $n=3$ for thread scaling), speedup differences smaller than $\sim$20\% should be interpreted with caution. Since speedup is the ratio of two medians, its uncertainty depends on the variability of both tools; with CV $<$ 5\% for most measurements, the headline speedup ratios (e.g., slope vs GDAL 1.8$\times$) are stable within $\pm$0.1--0.2$\times$ across the IQR.

GDAL's \texttt{gdaldem} slope/aspect/hillshade kernels are single-threaded by design---GDAL exposes no threading option for these operations (\texttt{GDAL\_NUM\_THREADS} governs warping and compression, not \texttt{gdaldem})---so its timings reflect the tool's only available configuration, not a handicap we imposed. GRASS~GIS \texttt{r.slope.aspect} supports OpenMP parallelism via \texttt{nprocs}; we report GRASS at both \texttt{nprocs=1} and \texttt{nprocs=16}. The comparison thus includes two further multi-threaded geocomputation frameworks beyond SurtGIS---WhiteboxTools (shared-memory) throughout, and TauDEM (MPI) for the hydrological algorithms (Section~\ref{subsec:taudem_mpi}).}

To ensure a fair comparison, every tool---including SurtGIS---is measured on the same end-to-end pipeline: read a GeoTIFF from disk, compute the result, and write the output GeoTIFF to disk. SurtGIS uses its native Rust GeoTIFF reader/writer (based on the \texttt{tiff} crate), with no GDAL dependency. GDAL is invoked via its \texttt{gdaldem} CLI; WhiteboxTools via its binary; GRASS~GIS via \texttt{grass --tmp-location}. All input DEMs are stored as uncompressed Float32 striped GeoTIFFs (no tiling, no compression) to eliminate codec-related variance and ensure the comparison reflects algorithmic performance rather than I/O library optimization. This is an important caveat: real-world DEMs are typically LZW- or DEFLATE-compressed, and SurtGIS's pure-Rust \texttt{tiff} crate is significantly slower than GDAL's optimized C codecs for decompression; the reported speedups should therefore be interpreted as upper bounds that may decrease in production workflows involving compressed data. We additionally report the SurtGIS compute-only time (excluding I/O) to quantify the algorithmic contribution.

GRASS exceeds the 300\,s timeout for fill at 5\,K$^2$ and above.

The I/O breakdown (Table~\ref{tab:all_algos}, Compute vs Full columns) reveals that file I/O accounts for 35--65\% of SurtGIS's total time for simple derivatives, compared to $<$10\% for computationally intensive algorithms such as fill and flow accumulation. SurtGIS's native Rust GeoTIFF reader/writer based on the \texttt{tiff} crate achieves competitive throughput for uncompressed data but does not implement the streaming I/O and SIMD codecs available in GDAL's C library.

\paragraph{I/O architecture} SurtGIS and WhiteboxTools load the entire raster into memory (read$\rightarrow$compute$\rightarrow$write), while GDAL and GRASS use streaming I/O that overlaps reading, computation, and writing. The ``Compute'' columns in Tables~\ref{tab:slope_results}--\ref{tab:all_algos} isolate the algorithmic contribution. The in-memory model favors repeated analyses on the same dataset (as in the case study, Section~\ref{sec:casestudy}), while streaming is preferable when dataset size approaches available RAM (Supplementary Table~S2).

For TPI, GDAL computes a simple $3 \times 3$ neighborhood difference, while SurtGIS uses a circular window of radius 10 cells ($\sim$314 cells per pixel); multi-scale TPI is preferred for geomorphological analysis \citep{dereu2013, newman2018}.

\subsubsection{Memory usage}

Peak memory usage was measured with \texttt{/usr/bin/time -v} on Linux at $10\text{K}^2$ resolution (Supplementary Table~S2). SurtGIS's default in-memory model requires 2.6\,GB peak RSS for slope (vs.\ 435\,MB for GDAL's streaming I/O), reflecting a $\sim$6.8$\times$ inflation factor over the raw 382\,MB Float32 input. This would limit the in-memory model to approximately $6{,}500^2$ cells on 8\,GB systems and $9{,}200^2$ cells on 16\,GB systems. The v0.7 release additionally includes out-of-core streaming variants for twelve terrain algorithms (slope, aspect, hillshade, TPI, TRI, roughness, classical curvatures, geomorphons, multi-scale curvature, sky-view factor, openness, MRVBF; see Section~\ref{subsec:parallel_strategies}) that hold a single row strip in RAM at a time, lifting this size limit at a modest throughput cost.

\subsection{Experiment 2: Numerical accuracy}
\label{subsec:accuracy}

\subsubsection{Analytical validation}

To validate numerical correctness, we compared SurtGIS outputs against closed-form analytical solutions on a Gaussian hill surface:

\begin{equation}
z(x,y) = A \exp\!\left(-\frac{x^2 + y^2}{2\sigma^2}\right)
\label{eq:gaussian}
\end{equation}

\noindent with $A = 500$\,m, $\sigma = 2000$\,m, on a $1000 \times 1000$ grid with 10\,m cell spacing. The analytical slope, aspect, and curvatures were derived from the closed-form partial derivatives of Eq.~\ref{eq:gaussian}. Boundary cells (1-pixel border) were excluded from comparison. Note that this smooth, infinitely differentiable surface represents a best-case scenario for finite-difference methods; real DEMs with sensor noise and terrain discontinuities will exhibit larger discretization errors. The cross-tool validation on a real DEM (Section~\ref{subsec:accuracy}) provides a complementary assessment under realistic conditions.

Table~\ref{tab:accuracy} presents the first-order results. SurtGIS achieves high numerical accuracy: slope RMSE of $5.7 \times 10^{-5}$~degrees, aspect maximum angular error of $2.2 \times 10^{-4}$~degrees, and $R^2 = 1.000000$ for both curvatures.

Validation of all 14 Florinsky curvatures on the same Gaussian hill (Supplementary Table~S4) confirms RMSE below $4 \times 10^{-9}$ for all types, with 11 of 14 yielding $R^2 = 1.000000$ (six decimal places). The three quantities that are analytically zero on radially symmetric surfaces ($k_{ve}$, $K_r$, rotor) show RMSE $< 10^{-9}$, confirming negligible absolute error. Eight Florinsky algebraic identities \citep{florinsky2025}---including $k_{\min} = H - M$, $k_{\max} = H + M$, $K = H^2 - M^2$---hold to within $1.6 \times 10^{-8}$.

Because the Gaussian hill is radially symmetric, three curvatures ($k_{ve}$, $K_r$, rotor) are identically zero and thus cannot be tested for proportional accuracy. To address this limitation, we repeated the validation on an anisotropic sinusoidal ridge $z = A \sin(k_x x) \cos(k_y y)$ with $A = 500$\,m, $\lambda_x = 4000$\,m, $\lambda_y = 3000$\,m on a $2000 \times 2000$ grid (5\,m cells), where all 14 curvatures are non-trivial. Excluding the bottom 10\% of gradient magnitude---a threshold chosen because gradient-dependent curvatures ($k_h$, $k_v$, $k_{he}$, $k_{ve}$, $K_a$, $K_r$, rotor) contain $\|\nabla z\|$ in their denominators and become numerically singular as $\|\nabla z\| \to 0$; results are stable for thresholds between 5\% and 20\%---all 14 curvatures achieve $R^2 = 1.000000$ with maximum RMSE $= 5.9 \times 10^{-8}$, confirming numerical correctness of the complete Florinsky system on non-symmetric terrain.

Cross-validation against SAGA~GIS (Evans method; seven overlapping curvatures; Supplementary Table~S5) revealed systematic methodological differences: the Laplacian agrees well (RMSE $= 2.5 \times 10^{-7}$\,m$^{-1}$, $R^2 = 0.9999$), confirming correct second-derivative computation, while gradient-dependent curvatures ($k_h$, rotor) diverge due to different formulations of slope-normalized terms \citep{minar2020}. These differences confirm that SurtGIS implements the Florinsky~(2025) formulation rather than the SAGA variant.

\paragraph{In-browser WASM vs server-based (B/S) deployment} Beyond raw compute time, the deployment model itself differs between the two ways a user can run a SurtGIS computation from a browser. Table~\ref{tab:wasm_vs_bs} contrasts the in-browser WebAssembly execution path with a classic Browser/Server (B/S) architecture (e.g.\ a JavaScript front-end such as Vue calling a Java/Spring Boot service that itself wraps GDAL or GRASS). The end-to-end user-perceived latency in the B/S case includes data upload, server queue and execution, and result download, which for a 4\,MB compressed GeoTIFF over a typical 10\,Mbps residential uplink is $\sim$3.2\,s of network time alone, before any compute. The in-browser WASM path eliminates these phases entirely: the user selects a file via the local \texttt{File} API and the result raster is materialised in browser memory without any HTTP round-trip. For DEMs that fit in browser memory ($\lesssim 2\text{K}^2$ in current Wasm builds, limited by the 32-bit linear memory addressing), this is decisive even though server-side compute on a dedicated CPU may be faster in absolute terms. The trade-offs invert at scale: B/S architectures amortise infrastructure across many tenants and can process DEMs too large for client RAM, but require server provisioning, authentication, queueing, and incur per-tenant compute cost.

A representative numeric comparison: a $1\,000 \times 1\,000$ slope computation in the browser takes 0.13\,s on V8 (Table~\ref{tab:wasm_1k}); the same operation on a typical t3.micro AWS instance running GDAL takes $\sim$0.10\,s, but adds upload of a $\sim$4\,MB GeoTIFF ($\sim$3.2\,s on a 10\,Mbps uplink) and download of the slope output. End-to-end the in-browser path is therefore $\sim$25$\times$ faster from the user's perspective on this size class, in addition to the privacy and offline-capability benefits. The crossover point at which the B/S model wins on latency depends on uplink speed and server provisioning, but for the interactive sub-second use cases that motivate a browser GIS (visual exploration, parameter sweeps, teaching tools), the in-browser model is dominant on the inputs it can hold in memory.

\subsection{Experiment 4: Parallel scaling analysis}
\label{subsec:parallel_scaling}

The scalability experiment in Section~\ref{subsec:scalability} compares SurtGIS to other tools at default thread settings, but does not isolate how SurtGIS itself scales with thread count. We address this here with a dedicated experiment: speedup $S(p) = T(1)/T(p)$ and parallel efficiency $E(p) = S(p)/p$ as a function of the number of threads $p$, for five algorithms representative of distinct parallel-decomposition regimes.

\paragraph{Methodology} We measure compute-only time (load and write are excluded from $T(\cdot)$) for slope, aspect, hillshade, Priority-Flood depression filling, and D8 flow accumulation on $5\text{K}^2$ (25\,M cells) and $10\text{K}^2$ (100\,M cells) synthetic fBm DEMs (same generator as Experiment~1). Thread counts $p \in \{1, 2, 4, 8, 12, 16\}$ are imposed via Rayon's \texttt{ThreadPoolBuilder::num\_threads(p).build\_global()}. Because Rayon's global pool can be initialised only once per process, the benchmark binary is re-launched for each $p$. Each $(\text{algorithm}, \text{size}, p)$ cell is run with 2 warm-up iterations followed by 5 measured repetitions; we report the median compute time. The hardware is the same Intel~i7-1270P configuration described in Section~\ref{sec:performance}: a hybrid topology of 4 performance cores (4.8\,GHz, 2 SMT threads each) and 8 efficiency cores (3.5\,GHz, single-threaded), totalling 16 logical threads. The hybrid layout is non-ideal for clean speedup curves but is representative of contemporary consumer hardware; we discuss its impact below.

\paragraph{Three scaling regimes} The data partition cleanly into three regimes that match well-established parallel-raster theory \citep{guan2010prpl, miao2017prplgtiol}:

\begin{enumerate}
    \item \emph{Memory-bandwidth-bound focal kernels.} Slope, aspect, and hillshade are embarrassingly parallel by row strip with a fixed-radius halo; in principle, $E(p) \approx 1$ should be achievable. In practice, $S(16)$ plateaus between $2.5$ and $4.0$ on both DEM sizes. The ratio between observed and ideal scaling is consistent with the per-thread memory bandwidth requirement saturating the system's DDR4-3200 channel: at $f64$ precision, slope reads nine input cells per output cell ($\approx 72$\,B per cell), and at peak compute throughput each thread requires $\sim$1.5\,GB/s, so 16 threads collectively demand $\sim$24\,GB/s on top of write traffic, a substantial fraction of the $\sim$50\,GB/s effective single-channel DDR4 bandwidth. Beyond 8 threads the curves flatten or, for hillshade at $5\text{K}^2$, mildly regress (3.21$\times$ at $p{=}12$, 2.91$\times$ at $p{=}16$): efficiency-cores entering the pool contribute throughput at roughly half the per-thread rate of performance-cores, while consuming a proportional share of the saturated memory bandwidth. This is fundamental to single-pass kernels on memory-constrained hardware rather than a limit of the SurtGIS implementation; on multi-channel server platforms (e.g.\ NUMA Xeon or EPYC), the same kernels would likely scale further before saturating.

    \item \emph{Amdahl-bound algorithms.} Priority-Flood depression filling \citep{barnes2014} relies on a single global priority queue: $S(16) \approx 1.05$ across both DEM sizes, indicating an effectively serial computation. Parallel variants exist---tile-decomposition schemes with boundary reconciliation \citep{barnes2017} are an obvious candidate---and integrating one is on the SurtGIS roadmap, but the present implementation prioritises algorithmic simplicity and exact result equivalence with the reference. The benchmark exposes this honestly: $E(p) \to 1/p$ is the expected behaviour of a sequential algorithm under thread sweeping and confirms there is no spurious parallelism inflating the reported $T(1)$.

    \item \emph{Dependency-aware partial parallelism.} D8 flow accumulation must respect the topological order of the flow DAG. SurtGIS's implementation processes cells in an order derived from the DAG's reverse-postorder, with limited per-stripe concurrency on independent sub-trees, yielding $S(16) \approx 1.2$--$1.5$. The scaling is sub-linear because most of the dependency structure on a single connected DEM lies on the critical path; a tile-and-reconcile strategy analogous to that of \citet{barnes2017} for fill would lift this ceiling, again without changing the per-cell algorithm.
\end{enumerate}

\paragraph{Load balancing} Because Rayon uses work-stealing scheduling \citep{blumofe1999} and the row-strip decomposition assigns roughly equal cell counts per worker, the three focal-kernel curves track each other within experimental noise across $p$. The departure from ideal $E$ visible in Figure~\ref{fig:scaling_efficiency} is therefore overwhelmingly attributable to memory-bandwidth saturation and hybrid-core heterogeneity rather than to load imbalance per se: a perfect-balance run on the same hardware would still hit the same bandwidth ceiling. The Amdahl-bound regime (Priority-Flood) is also a load-balance question only in a degenerate sense, because the work itself is sequential.

\paragraph{Implications for users} The headline scaling claims in Section~\ref{subsec:scalability} (e.g.\ $1.7$--$2.0\times$ over GDAL on slope) are obtained at default Rayon thread settings on the same hardware, so they implicitly include the bandwidth ceiling characterised here. Users running SurtGIS on multi-channel server platforms can therefore expect equal or better speedups than reported, while users restricted to a small number of cores (e.g.\ a 4-core laptop) can read off the expected slowdown directly from Table~\ref{tab:scaling}: $T(p)$ for $p=4$ on $10\text{K}^2$ slope is $771$\,ms, well within interactive use.

\subsection{\revtext{Experiment 5: Comparison against an MPI framework}}
\label{subsec:taudem_mpi}

\revtext{Experiments~1 and~4 compare SurtGIS to shared-memory tools. To position SurtGIS against the distributed-memory paradigm directly, we benchmark it against TauDEM~\citep{tarboton2015taudem}, a mature MPI-based parallel geocomputation framework that implements the same flow-routing algorithms (D8 and D-infinity flow direction and accumulation), with comparable depression-filling functionality. Both tools run on the identical single node (Intel i7-1270P, 16 hardware threads); TauDEM is launched under \texttt{mpiexec} with $W \in \{1,2,4,8,16\}$ processes and SurtGIS with the matching \texttt{RAYON\_NUM\_THREADS}$=W$, on the same DEMs. We report this single-node comparison with an explicit caveat: it cannot exhibit TauDEM's principal design advantage, scaling \emph{across} machines, which is irrelevant to the desktop, laptop, browser, and single-VM deployment surfaces SurtGIS targets but central to cluster workloads.}

\paragraph{\revtext{Protocol}} \revtext{A methodological asymmetry must be controlled. TauDEM's \texttt{pitremove} fills depressions to flat surfaces, which its flow-direction stage then resolves through an iterative flat-routing pass; SurtGIS's Priority-Flood fill imposes an $\varepsilon$ drainage gradient and leaves no flats. Comparing the two fill semantics directly would therefore benchmark flat-resolution policy rather than parallel-computation efficiency. We instead adopt a stage-wise protocol: the fill stage is timed per tool on the raw DEM (semantics differ, reported separately), while flow direction and accumulation are timed for both tools on a \emph{single shared} $\varepsilon$-filled DEM, so each stage compares identical computational work. The flat-routing observation is itself reported as a finding. Feeding TauDEM its own \texttt{pitremove} output instead of the shared $\varepsilon$-filled DEM would only \emph{increase} its flow-direction time (the uncorrected, flats-included pipeline timing is retained in the supplementary material), so the shared-input protocol is conservative toward SurtGIS; the $\varepsilon$-gradient fill that leaves no flats is a SurtGIS design choice, not a cost removed from the comparison.}

\paragraph{Position relative to service-publication frameworks} A complementary class of tools targets the \emph{publication} of geospatial computations as networked services rather than the in-process execution model emphasised here. The OGC API---Processes specification \citep{ogcapi_processes2024} defines a REST contract for invoking server-side geoprocessing, and reference implementations include GeoServer WPS \citep{geoserver_wps}, pygeoapi \citep{pygeoapi2023}, and PyWPS \citep{pywps2025}. Managed services such as the Sentinel Hub Processing API \citep{sentinelhub_api} and the Open Data Cube \citep{killough2018odc} expose curated raster archives with analytic primitives over HTTP. SurtGIS occupies a different rung of the same stack: it is the \emph{engine} of computation, not the \emph{protocol} of publication. A SurtGIS algorithm can be wrapped behind an OGC API---Processes adapter and made remotely invocable, and conversely SurtGIS can be called from any of the above frameworks via the Python binding. The two models trade off latency against scalability: the in-process model eliminates the network round-trip and serialisation overhead that dominate short-running terrain operations on small DEMs, particularly under the WebAssembly target where the data never leaves the client (Section~\ref{subsec:crossplatform}); the service-publication model amortises infrastructure across many tenants and avoids requiring users to install a runtime locally. We therefore view managed-service platforms as complementary, not competitive, with the SurtGIS computational core.

\section{Discussion and conclusions}
\label{sec:conclusions}

SurtGIS provides a high-performance, cross-platform geospatial analysis library that bridges the gap between computational efficiency and deployment flexibility. The systematic benchmarks presented in this paper demonstrate consistent performance advantages on fair end-to-end GeoTIFF pipelines at operationally relevant scales ($\geq$25\,M cells): 1.7--1.8$\times$ over GDAL for slope and 1.5--2.0$\times$ for aspect, 3.5--4.9$\times$ over GRASS, and 3.2--7.9$\times$ over WhiteboxTools for aspect and hillshade (the WBT slope comparison is excluded because WBT uses a different derivative method; see Section~\ref{subsec:accuracy}). For hydrological algorithms, SurtGIS achieves 1.6--2.6$\times$ speedups for depression filling and 7.5--23.1$\times$ for D8 flow accumulation. GDAL retains an advantage for hillshade at $\geq$5\,K$^2$, likely due to SIMD-optimized trigonometry. Numerical accuracy is within $5.7 \times 10^{-5}$~degrees RMSE against analytical solutions, and the Python binding overhead of 2.7--3.9$\times$ relative to native Rust is competitive with single-threaded native execution.

Several design decisions and limitations merit discussion. First, SurtGIS operates by default on in-memory arrays rather than streaming from disk, which limits the in-memory model to available RAM (peak RSS 1.9--2.6\,GB at $10\text{K}^2$; Supplementary Table~S2); the v0.7 release ships out-of-core streaming variants for twelve terrain algorithms (Section~\ref{subsec:parallel_strategies}), and broader streaming coverage is planned for future versions. Second, the TPI comparison highlights the importance of specifying algorithmic parameters when benchmarking: GDAL's $3 \times 3$ TPI and SurtGIS's radius-10 TPI are fundamentally different operations. Third, GDAL outperforms SurtGIS for hillshade at large DEM sizes, likely due to SIMD-optimized trigonometry; exploiting Rust's \texttt{std::arch} SIMD intrinsics is a targeted optimization opportunity.

The benchmarks have several limitations that should be considered when interpreting results. All measurements were collected on a single hardware platform (Intel i7-1270P with a heterogeneous architecture: 4 performance cores and 8 efficiency cores) under the default \texttt{powersave} CPU governor with Turbo Boost enabled. Speedup ratios may differ on other architectures (e.g., AMD Ryzen, ARM64 Apple Silicon) or systems with homogeneous core configurations, where Rayon's work-stealing scheduler may exhibit different scaling behavior. The scalability experiment uses synthetic fractal Brownian motion DEMs; real-world DEMs with irregular NoData patterns, extensive flat regions, or sensor artifacts may produce different performance characteristics, particularly for depression filling and flow routing algorithms. The use of uncompressed Float32 GeoTIFFs eliminates codec-related variance but does not represent typical workflows where DEMs are LZW- or DEFLATE-compressed; SurtGIS's pure-Rust \texttt{tiff} crate decoder is significantly slower than GDAL's optimized C codecs for compressed data, which would reduce the observed speedup advantage in production settings.

The open-source implementation of Florinsky's complete 14-curvature morphometric system \citep{florinsky2025} removes a barrier to adoption of these theoretically superior curvature descriptors in environmental modeling. The integrated hydrological pipeline---from depression handling through flow direction to derived indices---provides a cohesive workflow without requiring users to chain multiple tools.

\paragraph{Validation coverage} The benchmarks and cross-tool comparisons presented here cover the most commonly used algorithms: slope, aspect, hillshade, depression filling (Priority-Flood), D8 and D-infinity flow direction, and flow accumulation. The 14 Florinsky curvatures are validated analytically on two surfaces (Gaussian hill and anisotropic sinusoidal ridge) and cross-validated against SAGA for the seven overlapping types. Several algorithms listed in Supplementary Table~S1 have not been independently validated in this paper, including: MFD flow accumulation (only D8 was cross-compared), TFGA flow direction, Planchon--Darboux filling (only Priority-Flood was benchmarked), watershed delineation, and Strahler stream ordering. These algorithms pass internal unit tests (637 total across the test suite) but have not been compared against external reference implementations. Systematic cross-tool validation of these remaining algorithms is planned for future work.

Within the Python hydrological ecosystem, SurtGIS occupies a complementary niche. pysheds \citep{bartos2020pysheds} provides lightweight D8/D-infinity routing with efficient memory use, and pyflwdir \citep{eilander2021pyflwdir} offers fast pure-Python flow direction methods compatible with HydroSHEDS and MERIT Hydro datasets; both are well suited for interactive notebook workflows. SurtGIS offers a broader scope (terrain morphometry, depression handling, and derived indices alongside flow routing) and significantly higher throughput for large DEMs through compiled Rust code, but currently lacks native support for standard hydrological datasets such as HydroSHEDS or MERIT Hydro; users must load these as GeoTIFF via external tools. The \texttt{surtgis-cloud} crate provides experimental STAC catalog browsing and COG reading via HTTP range requests, but cloud-native hydrological workflows (e.g., streaming watershed analysis over COG tiles) are not yet implemented and remain a development target.

The cross-platform architecture enables new use cases for environmental modeling: browser-based teaching tools for terrain analysis, client-side processing of sensitive data that cannot leave the user's machine, and offline-capable field applications via Progressive Web Apps. However, the current WASM target is single-threaded, which limits practical browser-based processing to DEMs up to approximately $1\text{K}^2$--$2\text{K}^2$ cells for interactive use; a complete hydrological workflow (fill + flow direction + accumulation + TWI) on a $5\text{K}^2$ DEM would take $>$15\,s in the browser (extrapolated from Table~\ref{tab:crossplatform}). The Python bindings make SurtGIS accessible to the large scientific Python ecosystem while preserving the performance benefits of the native Rust implementation.

\paragraph{Software sustainability} SurtGIS is currently developed and maintained by a single author (latest release v0.7.0 at the time of writing), which raises legitimate concerns about long-term sustainability compared to established projects such as GDAL ($>$100 contributors, 20+ years of development) or GRASS GIS (active since 1982). To mitigate the single-developer risk: (a)~the project uses permissive dual licensing (MIT/Apache-2.0) to lower the barrier for external contributions; (b)~continuous integration enforces code quality across 9 automated jobs including cross-platform builds; (c)~the modular crate architecture allows contributors to work on individual components without understanding the full codebase; and (d)~the Rust API follows semantic versioning, with the 0.x series indicating that breaking changes may occur before a 1.0 stability commitment. Community adoption is in its early stages; building a contributor base and establishing governance structures are priorities for the project's next phase.

Future development will focus on: (1) expanding WebAssembly support to include multi-threaded execution via SharedArrayBuffer; (2) extending streaming I/O coverage to the hydrology and imagery modules (the v0.7 release ships streaming variants for twelve terrain algorithms; broader coverage will lift the in-memory size limit for the remaining algorithm classes); (3) parallel tile-and-reconcile variants of Priority-Flood and D8 flow accumulation following \citet{barnes2017} to break the Amdahl ceiling identified in Experiment~4; (4) GPU acceleration for computationally intensive operations via WebGPU; and (5) maturing the STAC + COG cloud workflow stack toward streaming watershed analysis directly over COG tiles.
